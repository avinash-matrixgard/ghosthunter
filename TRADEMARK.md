## What this means in practice

### You CAN

- Fork the repository
- Modify the code under AGPL-3.0 terms
- Run modified versions internally
- Run modified versions as a public service (with the AGPL-3.0
  source-disclosure obligations that come with that)
- Reference Ghost-hunter by name in articles, blog posts, comparisons,
  reviews, talks, academic papers — all standard nominative fair use
- Say "this fork is based on Ghost-hunter" in your own README, as long
  as you don't imply endorsement by MatrixGard

### You CANNOT

- Distribute a fork under the name "Ghost-hunter" or any
  confusingly-similar name (e.g. "GhostHunter", "Ghost-Hunter Pro",
  "Ghosthunter+")
- Use "Ghost-hunter" or any associated logos in your own product's
  brand identity
- Suggest that your fork is the official Ghost-hunter or that
  MatrixGard endorses it
- Register domain names containing "ghosthunter" / "ghost-hunter" with
  intent to capitalize on this project's reputation

### If you fork

Please rebrand. Pick your own name for your fork. This is standard
practice — see how forks of MongoDB became "DocumentDB", forks of
Elasticsearch became "OpenSearch", forks of MySQL became "MariaDB".
